# Benchmark Datasets

This document records which real-world datasets are used or planned for
SparrowDB benchmarking, where to obtain them, how to load them, and what
each dataset validates that synthetic graphs cannot.

### Email-Enron

| Property | Value |
|----------|-------|
| Nodes | 36 692 |
| Edges | 183 831 |
| Type | Directed email communication graph |
| Source | https://snap.stanford.edu/data/email-Enron.html |
| Format | Edge list (tab-separated) |
| Why | ~10× SNAP FB; tests CSR scaling; real-world directed graph |

**What this validates that SNAP FB doesn't:**
- B-tree property index at 36K nodes (depth increases)
- `execute_two_hop` result set size at ~10× edge count
- WAL performance with larger node count on cold open

**Load command (once importer supports SNAP edge-list format):**
```bash
sparrowdb bulk-import \
  --db enron.db \
  --edges Email-Enron.txt \
  --rel-type SENT_TO \
  --src-label Person --dst-label Person
```

### MovieLens 25M

| Property | Value |
|----------|-------|
| Nodes | ~9 000 users + ~62 000 movies |
| Edges | 25 000 095 rating edges |
| Type | Bipartite user–movie rating graph |
| Source | https://grouplens.org/datasets/movielens/25m/ |
| Format | CSV (userId, movieId, rating, timestamp) |
| Why | AI recommendation use case; tests edge properties at scale |

**What this validates:**
- Edge properties at 25M edges (rating value, timestamp)
- Bipartite traversal patterns: `MATCH (u:User)-[:RATED {rating: 5}]->(m:Movie)`
- Memory and WAL performance during 25M-edge bulk load
- Whether bulk-import throughput degrades at this volume

**Load sketch:**
```bash
# Convert to node CSV + edge CSV, then:
sparrowdb bulk-import \
  --db movielens.db \
  --nodes users.csv --node-label User \
  --nodes movies.csv --node-label Movie \
  --edges ratings.csv --rel-type RATED \
  --src-label User --dst-label Movie \
  --batch-size 50000
```

## Why Larger Datasets Matter Beyond "More Rows"

### Buffer pool pressure

The 4K-node SNAP graph fits in ~1MB.  Amazon (262K nodes) needs ~50MB.
LDBC SF1 needs ~500MB.  When data exceeds L3 cache (~32MB on M-series),
the OS page cache becomes the bottleneck and cold-read latency dominates.
This reveals whether `madvise(SEQUENTIAL)` hints (commit `e4c8e02`) are
sufficient or whether an explicit buffer pool is needed.

### CSR index depth and fan-out distribution

Real social graphs follow a power-law degree distribution: most nodes
have few edges, a few "hub" nodes have thousands.  The synthetic ring
graph has uniform fan-out (20 per node).  Power-law graphs stress the
CSR reader differently — hub nodes produce very large adjacency lists
that don't fit in a single CSR chunk.

### B-tree property index depth

At 4K nodes the B-tree property index is at most depth 2.  At 300K nodes
it reaches depth 4–5.  This exercises different code paths in
`PropertyIndex::lookup` and can reveal cache-line thrashing on interior
node traversals.

### WAL segment rotation

The WAL rolls over to a new segment file after a configurable size
threshold.  Small datasets never trigger rollover.  Large bulk loads
(25M edges) exercise segment creation, the `active_segment` pointer
update, and crash recovery across segment boundaries.

### Algorithm complexity validation

A performance characteristic that looks O(N log N) at 4K nodes may
actually be O(N²) with a small constant — invisible until 400K nodes.
Running the full suite against Email-Enron and Amazon confirms that no
query path has accidentally quadratic behaviour at practical scale.

## Adding a New Dataset

1. Add an entry to this file under "Planned Datasets" with source,
   format, and what it validates.
2. Write a loader script or `bulk-import` command and test it.
3. Add a criterion benchmark function in `snap_benchmarks.rs` or a new
   `<dataset>_benchmarks.rs` file covering the representative query.
4. Run baseline on `main`, record the number in this file.
5. Move the entry to "Currently Used" and add the baseline row to the
   table at the top.
6. If the dataset covers a query in the README benchmark table, run on
   it with 50 warmup + 200 iterations and update the table.
